# 《第一批工程包任务单》

> 文档属性：内部执行稿 / 首批正式派工母表  
> 适用范围：`gongwen-assistant` 项目 P0-P1 启动阶段  
> 编制依据：`project-charter.md`、`workstreams.md`、`agents.md`、`dispatch-playbook.md`、`task-card-schema.md`，并结合《语料入库规范》《文种清单与优先级》补齐首批落地任务  
> 文档目标：把项目从“有制度文档”推进到“有明确工程包、责任归属、依赖关系、验收口径、下一步动作”的正式执行阶段。

---

## 一、编制原则

本批任务单遵循以下原则：

1. **先做闭环底座，再做能力扩张**：优先建设任务卡、规则中枢、模板骨架、语料入库、工作流/表单等底层能力。
2. **先做可执行工程包，不做空泛规划**：每个任务必须能直接派发到责任省部继续推进。
3. **起草、审议、执行分离**：严格遵循三省六部制度，避免单点黑箱推进。
4. **验收必须可核对**：每个任务都给出清晰产物、验收标准、依赖与后续动作。
5. **首批覆盖四个方向**：规则中枢、模板骨架、语料入库、工作流/表单为本批硬覆盖范围。

## 三、正式任务单

## GW-EB-001｜建立规则中枢 V1（文种规则 + 审校规则 + 红线规则）

- **任务类型**：规范/规则工程包
- **责任省部**：**门下省牵头，礼部 / 刑部 / 工部协同**
- **对应方向**：规则中枢
- **任务目标**：
  - 把现有《项目总方案》《文种清单与优先级》《语料入库规范》中的约束沉淀为首版可执行规则中枢。
  - 形成“生成前约束 + 审校时校验 + 红线项阻断”的统一规则资产，为模板、审校、工作流提供上位依据。
- **主要产物**：
  1. 首批文种规则清单（通知、请示、报告、工作方案优先）
  2. 审校规则分级清单（阻断项 / 高优项 / 一般项 / 观察项）
  3. 红线规则清单（伪造依据、伪造数据、越权口径、无来源强结论等）
  4. 规则版本与维护责任表
- **依赖**：无
- **验收标准**：
  1. 至少覆盖 4 个 P1 文种的结构规则、标题规则、结尾习惯、禁用项。
  2. 至少形成 20 条以上结构化审校规则，并完成分级。
  3. 红线规则明确可拦截的高风险情形，不与上位文档冲突。
  4. 规则项可被后续模板库、审校引擎、表单提示共同引用，而不是只有散文式说明。
- **下一步**：
  1. 将规则中枢输出为统一目录或结构化文件草案。
  2. 向 GW-EB-002（模板骨架）和 GW-EB-005（语料入库）同步规则口径。
  3. 由工部评估后续 JSON/YAML 化方案。

## GW-EB-002｜建立首批模板骨架 V1（四大高频文种）

- **任务类型**：模板工程包
- **责任省部**：**中书省牵头，礼部协同**
- **对应方向**：模板骨架
- **任务目标**：
  - 基于文种优先级，先把首批最关键的起草骨架做出来，确保系统能够稳定输出“像公文的首稿框架”。
  - 首批至少覆盖：通知、请示、报告、工作方案。
- **主要产物**：
  1. 每个文种的标准骨架模板
  2. 子类型模板（如通知的部署类/会议类/报送类/印发类）
  3. 待补信息提示项清单
  4. 模板使用说明：适用场景 / 不适用场景 / 常见误用
- **依赖**：GW-EB-001
- **验收标准**：
  1. 4 个高频文种均有标准骨架，且字段命名与任务卡规范兼容。
  2. 每个文种至少列出 1-3 个高频子型模板。
  3. 每个模板明确“必备结构段、可选结构段、禁止混写项”。
  4. 礼部完成至少一轮文面与体例复核，确认模板不明显“像文章不像公文”。
- **下一步**：
  1. 与 GW-EB-006 联动，用首批优质样本和片段补强模板内容。
  2. 与 GW-EB-007 联调，检验模板在起草—审校链路中的稳定性。
  3. 在第二批扩展函、会议纪要。

## GW-EB-003｜建立任务卡与表单字段映射 V1

- **任务类型**：结构化表单工程包
- **责任省部**：**吏部牵头，尚书省 / 工部协同**
- **对应方向**：工作流/表单
- **任务目标**：
  - 以《任务卡字段规范》为母版，建立“任务卡字段 → 表单字段 → 流转字段”的首版映射关系。
  - 解决后续任务接入时“描述不统一、字段不齐、系统与制度脱节”的问题。
- **主要产物**：
  1. 任务卡最小必填字段表单版
  2. 高级补充字段清单
  3. 自动生成字段与状态字段映射表
  4. 字段校验与触发规则（例如 need_review=true 时显示 review_focus）
- **依赖**：无
- **验收标准**：
  1. 最小必填集合完整覆盖任务理解、起草、审校、时限控制四类关键字段。
  2. 字段状态清晰区分：必填 / 选填 / 自动生成。
  3. 至少明确 10 条表单联动或校验逻辑。
  4. 表单字段与任务流转编号、角色、状态字段不冲突。
- **下一步**：
  1. 把表单映射交给 GW-EB-004 用于状态机与派工台账设计。
  2. 工部据此评估后续 JSON Schema 或数据表实现方式。
  3. 尚书省将其纳入正式开单模板。

## GW-EB-004｜建立任务流转状态机与派工台账 V1

- **任务类型**：流程编排工程包
- **责任省部**：**尚书省牵头，吏部 / 刑部 / 工部协同**
- **对应方向**：工作流/表单
- **任务目标**：
  - 以《任务流转清单与下旨模板》为基础，正式落地首版状态机、节点职责、派工台账口径。
  - 把任务从“写文档规则”推进到“能按状态流转、能留痕、能追责”。
- **主要产物**：
  1. 首版任务状态机定义
  2. 派工台账字段模板
  3. 下旨/送审/派工/驳回/验收标准单模板索引
  4. 任务升级、退回、阻塞上报规则
- **依赖**：GW-EB-003
- **验收标准**：
  1. 状态至少覆盖：待立项、待起草、待审议、待派发、进行中、待复核、已退回、已完成、已归档。
  2. 每个状态都明确谁可更新、何时进入、何时退出。
  3. 每项任务都能记录主责、协办、复核、截止时间、交付路径、最近更新时间。
  4. 形成最小闭环：开单 → 派工 → 执行 → 审校 → 验收 → 归档。
- **下一步**：
  1. 为 GW-EB-008 提供试运行底座。
  2. 由工部评估是否同步沉淀为脚本校验规则。
  3. 尚书省据此建立首批真实任务台账样例。

## GW-EB-005｜建立公开语料入库流水线 V1

- **任务类型**：数据治理工程包
- **责任省部**：**户部牵头，礼部 / 门下省 / 工部协同**
- **对应方向**：语料入库
- **任务目标**：
  - 以《语料入库规范》为上位约束，建立首批可执行的公开语料入库流程，先把“原件—清洗件—元数据—分层—审核”链路跑通。
  - 确保首批语料资产来源可追溯、结构可用、不会污染后续模板和规则库。
- **主要产物**：
  1. 公开语料来源白名单草案
  2. 入库流程卡（抓取、初筛、清洗、标注、复核、入库）
  3. 元数据字段模板
  4. 三层分层口径：原文层 / 规则层 / 优质片段层
- **依赖**：GW-EB-001
- **验收标准**：
  1. 明确一级来源、二级来源、禁入来源边界。
  2. 完成首版元数据必填字段模板，至少覆盖 corpus_id、source_url、publisher、publish_date、doc_type、status、reviewer 等关键字段。
  3. 明确 raw / clean / metadata / rules / snippets / review 六类资产的最小目录组织方式。
  4. 至少完成 1 轮入库演练说明，证明流程可执行。
- **下一步**：
  1. 直接支撑 GW-EB-006 的样本基准集建设。
  2. 工部据此评估抓取、去重、标注脚本的实现顺序。
  3. 门下省与刑部补一轮风险抽查意见。

## GW-EB-006｜建立首批文种样本基准集与优质片段库 V1

- **任务类型**：语料资产工程包
- **责任省部**：**户部牵头，礼部 / 中书省协同**
- **对应方向**：语料入库 + 模板骨架
- **任务目标**：
  - 面向首批高频文种，建立一组可供模板设计、提示词约束、审校对照使用的样本基准集和优质片段库。
  - 不追求大规模，而追求“高质量、小而准、能直接服务工程联调”。
- **主要产物**：
  1. 首批文种样本清单（通知、请示、报告、工作方案优先）
  2. 优质片段库（标题、开头、过渡、部署要求、结尾）
  3. 正反例说明
  4. 片段与原文绑定关系说明
- **依赖**：GW-EB-002、GW-EB-005
- **验收标准**：
  1. 每个首批文种至少沉淀 5 份可用样本或等价规模的基准材料。
  2. 每个文种至少抽取 8-10 条可复用优质片段，并标明适用章节。
  3. 所有片段均绑定原文来源，不允许无根摘句。
  4. 礼部确认片段表达符合文种规范，中书省确认片段对模板补强有实际价值。
- **下一步**：
  1. 将片段反馈给 GW-EB-002 优化模板骨架。
  2. 将样本集交给 GW-EB-007 做联调测试输入。
  3. 后续扩展到函、会议纪要、总结、简报。

## GW-EB-007｜建立起草—审校联调链路 V1

- **任务类型**：联调验证工程包
- **责任省部**：**中书省牵头，门下省 / 礼部 / 刑部协同**
- **对应方向**：规则中枢 + 模板骨架
- **任务目标**：
  - 把规则中枢和模板骨架真正串起来，验证系统能否形成“先按文种起草，再按规则审校”的最小联调闭环。
  - 核查首批模板是否能被规则有效约束，首批规则是否能对模板输出给出可执行审校意见。
- **主要产物**：
  1. 联调测试清单
  2. 文种级起草—审校对照样例
  3. 规则命中记录表
  4. 常见问题回收清单
- **依赖**：GW-EB-001、GW-EB-002
- **验收标准**：
  1. 至少完成 4 个高频文种各 1 轮联调。
  2. 每轮联调都能输出：首稿、审校意见、修改建议、问题归类。
  3. 至少识别出 10 个以上真实问题点，并能归因到模板缺口、规则缺口或任务输入缺口。
  4. 礼部、门下省、刑部三方对联调问题分类达成一致口径。
- **下一步**：
  1. 把联调发现回灌到 GW-EB-001 和 GW-EB-002。
  2. 为 GW-EB-008 提供首轮验收材料。
  3. 视结果决定第二批文种扩展顺序。

## GW-EB-008｜建立首轮试运行验收包与问题回收机制

- **任务类型**：验收/复盘工程包
- **责任省部**：**刑部牵头，尚书省 / 门下省 / 工部协同**
- **对应方向**：闭环验收
- **任务目标**：
  - 为首批工程包建立统一的试运行验收口径，确保后续不是“看起来做了很多”，而是“确实形成可用闭环”。
  - 形成问题回收与责任归因机制，为第二批工程包提供清晰输入。
- **主要产物**：
  1. 首轮试运行验收清单
  2. 正反例问题台账
  3. 问题归因分类表（语料 / 规则 / 模板 / 工作流 / 表单）
  4. 二期改进建议清单
- **依赖**：GW-EB-004、GW-EB-006、GW-EB-007
- **验收标准**：
  1. 验收维度至少覆盖任务理解准确性、内容生成质量、审校有效性、流程可控性、组织复用价值。
  2. 每个问题都能回溯到具体任务编号、具体环节、具体责任省部。
  3. 至少形成一版“通过 / 通过但需整改 / 不通过”的分级判定口径。
  4. 输出可直接作为第二批工程包输入的整改列表。
- **下一步**：
  1. 尚书省据此组织首轮结案与归档。
  2. 工部根据问题台账安排后续自动化、脚本化优先级。
  3. 形成第二批工程包候选项（函/会议纪要扩展、审校规则细化、语料脚本化入库等）。

## 五、责任分派摘要（便于尚书省后续下旨）

- **门下省**：主抓规则中枢、审议边界、风险口径
- **中书省**：主抓模板骨架、起草联调、输出结构稳定性
- **尚书省**：主抓状态机、派工台账、试运行组织与归档
- **吏部**：主抓任务卡、角色映射、表单字段规范化
- **户部**：主抓语料来源、元数据、样本基准集与片段库
- **礼部**：主抓文种体例、文面规范、模板与语料风格复核
- **刑部**：主抓规则稽核、验收清单、问题归因与整改闭环
- **工部**：主抓结构化落地、后续脚本化/Schema 化/自动化评估

## 六、一句话结论

第一批工程包不追求“把 gongwen-assistant 一次做完”，而是先把**规则中枢、模板骨架、语料入库、工作流/表单**四条主干正式立起来，再通过联调和验收把项目推入真正可执行、可留痕、可迭代的工程状态。
