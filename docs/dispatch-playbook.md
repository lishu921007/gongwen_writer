# 任务流转清单与下旨模板

> 适用范围：`gongwen-assistant` 项目的标准任务流转、督办、复核与归档。  
> 使用方式：接到任何新事项后，先按本文件完成“流转清单”，再使用对应“下旨模板”正式派发。

---

## 一、标准流转原则

1. **先立项编号，再进入执行。**
2. **先有牵头与复核，再下发承办。**
3. **未经审议的草案，不得直接视为定令。**
4. **任何正式任务不得由单一 Agent 从头包办到尾。**
5. **所有状态变化必须留痕，禁止口头流转。**

建议任务编号格式：

`GW-年份-月份-序号`  
示例：`GW-2026-03-001`

---

## 二、标准流转总图

```text
需求进入
  ↓
太子定性（是否立项 / 优先级 / 牵头方）
  ↓
中书省起草（目标、范围、拆解、产物、时限）
  ↓
门下省审议（风险、驳正、放行/退回）
  ↓
尚书省立单并派发（责任人、节点、状态）
  ↓
六部执行（按专业域承办）
  ↓
刑部/指定复核人验收
  ↓
礼部整理正式文面（如需对外/正式归档）
  ↓
尚书省归档与回报
  ↓
太子确认结案 / 进入下一轮
```

## 三、任务流转清单（逐项勾选）

以下清单建议每个任务开单时逐条确认。

### A. 立项前检查

- [ ] 是否明确任务来源（谁提出、为何提出）
- [ ] 是否明确任务目标（一句话说清要达成什么）
- [ ] 是否明确范围（做什么 / 不做什么）
- [ ] 是否明确交付物（文档、代码、报告、表格、发布稿等）
- [ ] 是否明确时限（硬截止 / 软截止）
- [ ] 是否判断优先级（高 / 中 / 低 / 紧急）
- [ ] 是否指定太子裁定结果（立项 / 暂缓 / 驳回）

### B. 分工前检查

- [ ] 是否指定中书省主起草人
- [ ] 是否指定门下省审议人
- [ ] 是否指定尚书省督办人
- [ ] 是否指定承办部门（六部之一或多个）
- [ ] 是否指定主责 Agent
- [ ] 是否指定复核 Agent
- [ ] 是否确认**不是单 Agent 包办**

### C. 执行前检查

- [ ] 是否已生成任务编号
- [ ] 是否已形成任务草案/正式单
- [ ] 是否补齐依赖与前置条件
- [ ] 是否写明风险点与回滚方式
- [ ] 是否说明汇报频率与节点
- [ ] 是否给出交付路径（目录、文件、仓库位置）

### D. 执行中检查

- [ ] 当前状态是否已更新（未开始 / 进行中 / 待复核 / 已完成 / 已退回）
- [ ] 阻塞项是否已上报尚书省
- [ ] 是否发生范围变更
- [ ] 是否需要太子重新裁定优先级
- [ ] 是否需要门下省补充审议

### E. 收尾检查

- [ ] 是否完成交付物
- [ ] 是否完成复核/验收
- [ ] 是否记录修改意见与返工结果
- [ ] 是否完成归档
- [ ] 是否输出结案说明
- [ ] 是否沉淀复盘要点

---

## 四、任务状态定义

| 状态 | 含义 | 由谁更新 |
|---|---|---|
| 待立项 | 需求已进入，尚未裁定是否启动 | 太子 / 尚书省 |
| 待起草 | 已立项，待中书省形成草案 | 中书省 |
| 待审议 | 草案完成，待门下省审核 | 门下省 |
| 待派发 | 已审定，待尚书省正式分派 | 尚书省 |
| 进行中 | 六部/承办 Agent 正在执行 | 承办方 |
| 待复核 | 执行完成，待刑部或指定复核人检查 | 复核方 |
| 已退回 | 审议或验收未通过，需返工 | 门下省 / 刑部 |
| 已完成 | 验收通过，待归档或已归档 | 尚书省 |
| 已归档 | 已留痕、可追溯、流程闭合 | 尚书省 |

## 五、标准流转动作

### 第 1 步：太子定性

太子需明确：

- 此事做不做
- 先后级别如何
- 谁牵头起草
- 是否属于高风险/高影响事项

**输出**：立项意向或裁定意见。

---

### 第 2 步：中书省起草

中书省需形成任务草案，至少包含：

- 背景
- 目标
- 范围
- 交付物
- 时限
- 分工建议
- 风险提示

**输出**：任务草案 / 下旨草案。

---

### 第 3 步：门下省审议

门下省重点审查：

- 目标是否清晰
- 分工是否合理
- 是否违反“不得单 Agent 包办”
- 风险是否被识别
- 验收标准是否可执行

**输出**：通过 / 修改后通过 / 驳回。

---

### 第 4 步：尚书省派发

尚书省在收到放行后：

- 分配主责与协办
- 设定时间节点
- 建立任务台账
- 明确汇报节奏
- 记录交付路径

**输出**：正式执行单。

---

### 第 5 步：六部执行

承办方按职责落实：

- 吏部：配岗与权限
- 户部：资源与台账
- 礼部：规范与文面
- 兵部：调度与应急
- 刑部：质检与问责
- 工部：开发与自动化

**输出**：阶段成果、问题清单、最终交付物。

---

### 第 6 步：复核与归档

- 刑部或指定复核人进行验收
- 礼部整理正式文面（如适用）
- 尚书省记录完成状态并归档
- 太子确认结案或进入下一轮指令

**输出**：验收结论、归档记录、结案说明。

## 六、标准下旨模板

以下模板可按场景直接复制使用。

### 模板 1：立项下旨（太子 → 中书省）

```markdown
# 立项下旨

- 任务编号：GW-YYYY-MM-XXX
- 任务名称：
- 来源：
- 裁定人（太子）：
- 裁定时间：

## 裁定内容
请中书省就以下事项起草正式执行草案：

- 背景：
- 目标：
- 范围：
- 预期交付物：
- 初步时限：
- 优先级：高 / 中 / 低 / 紧急

## 明确要求
- 不得由单一 Agent 包办全流程。
- 须明确主责、协办、复核三类角色。
- 涉及高风险事项需单列风险与回滚方案。

## 输出要求
请于【时间】前提交：
1. 任务草案
2. 分工建议
3. 风险清单
```

## 七、常用短指令模板（后续“下旨”可直接套用）

适用于聊天界面快速派发，再由中书/尚书转成正式单。

### 1. 给中书省的快令

```text
中书省即刻起草：围绕【事项】，在【时间】前提交任务草案。
须写清：目标、范围、交付物、分工建议、风险点。
强调：不得设计成单 Agent 包办流程。
```

### 2. 给门下省的快令

```text
门下省即刻审议：请核查【任务编号 / 事项】是否目标清晰、职责分离、风险充分。
请给出结论：通过 / 修改后通过 / 驳回，并写明理由。
```

### 3. 给尚书省的快令

```text
尚书省依审定结果正式派工：请明确牵头部门、主责 Agent、复核 Agent、时间节点、交付路径，并开始督办。
```

### 4. 给六部的快令

```text
【某部】承此事项：请按职责落实【具体事项】，于【时间】前提交【交付物】；如有阻塞，立即上报尚书省。
```

### 5. 给刑部的快令

```text
刑部对【任务编号】做最终核查：重点检查事实、逻辑、格式、留痕与职责分离。未经核查，不得标记结案。
```

## 九、执行提醒

1. **太子只定方向，不替代全员劳动。**
2. **中书只负责写明白，不直接把草案当圣旨。**
3. **门下必须敢于驳回问题草案。**
4. **尚书必须盯进度、盯交付、盯归档。**
5. **六部必须按专业分责，不越位、不缺位。**
6. **任何时候都不得回到“一个 Agent 全包”的低级模式。**

> 真正成熟的 `gongwen-assistant`，不是靠某一个 Agent 英雄主义冲刺，而是靠可复制、可审计、可持续的任务流转制度稳定产出。
