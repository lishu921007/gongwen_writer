# 公文助手（gongwen-assistant）项目总方案

> 文档属性：内部执行稿  
> 适用对象：主代理、三省六部执行单元、后续参与协作的子代理/人工审核者  
> 文档目标：统一项目目标、范围、分期、能力建设、语料治理、多 agent 制度与验收口径，避免“边做边猜”“各自为战”。

---

## 1. 项目定位

公文助手（gongwen-assistant）不是一个单点写作器，而是一套面向**公文生产、审校、归档、知识沉淀与协同流转**的内部智能执行系统。

它服务的核心场景不是“给外部看一篇漂亮文章”，而是帮助内部团队稳定完成以下闭环：

1. **接收任务**：识别公文场景、对象、文种、时效要求、口径约束。
2. **组织材料**：聚合政策依据、历史模板、既有稿件、业务事实、领导偏好。
3. **起草与改写**：生成符合文种要求、层次清晰、口径统一的草稿。
4. **审校把关**：检查格式、逻辑、事实、措辞、风险、敏感表达。
5. **版本流转**：支持多轮反馈、责任留痕、差异对比、定稿归档。
6. **知识沉淀**：把有效模板、典型表述、审校规则、案例经验沉淀为可复用资产。

一句话：**这是一个面向“内部公文生产体系化”的项目，不是单纯的 AI 写稿功能。**

---

## 2. 项目目标

## 2.1 总目标

建设一套可持续迭代的公文智能协作体系，使主代理能够统筹任务，三省六部能够分工执行，并最终形成：

- 可控的公文生成能力
- 可审的内容治理能力
- 可复用的语料与规则资产
- 可验收的流程化交付能力

## 2.2 阶段性业务目标

### 第一目标：先把“能稳定产出”做出来

优先解决以下问题：

- 接到任务后，不再从零开写
- 不同文种有可执行的起草框架
- 同一任务多轮改写时不丢上下文
- 可以快速给出“可编辑草稿”而不是空白页

### 第二目标：再把“质量稳定”做出来

重点解决：

- 文风漂移
- 套话堆积
- 逻辑不闭合
- 事实引用不清
- 与内部口径不一致
- 结构像文章、不像公文

### 第三目标：最后把“组织能力”做出来

形成：

- 语料分层管理
- 审校规则体系
- 多 agent 分工制度
- 版本与责任留痕
- 可迁移到更多文种/部门/场景的能力底座

## 2.3 成功标准（高层）

项目成功不以“模型说得多像人”为标准，而以以下标准衡量：

- 任务接收后可以快速进入草拟阶段
- 常见文种具备稳定模板与约束框架
- 输出文本经审校后可进入内部流转
- 关键问题可定位到责任环节（语料、规则、生成、审校、人工确认）
- 新任务越做越快，越做越稳，而不是每次重来

## 3. 项目边界

## 3.1 本项目要做什么

本项目聚焦以下边界内能力：

1. **公文相关文本任务**
   - 通知、请示、报告、汇报材料、会议纪要、讲话提纲、方案、总结、函、简报等
2. **与公文有关的结构化辅助能力**
   - 文种识别
   - 提纲生成
   - 素材抽取与归并
   - 口径统一
   - 改写压缩扩写
   - 风险审校
   - 定稿前清单检查
3. **围绕公文业务的知识沉淀**
   - 模板库
   - 表述库
   - 语料库
   - 审校规则库
   - 场景 SOP

## 3.2 本项目暂不做什么

以下内容默认不纳入首期硬目标：

1. **全领域通用大模型平台化建设**
   - 不追求覆盖所有办公写作任务
2. **面向公众的开放式产品化发布**
   - 当前是内部执行体系，不是外部 SaaS
3. **高风险自动发送/自动盖章/自动对外发文**
   - 首期只做到“辅助起草 + 审校 + 流转建议”，不做无人值守发文
4. **脱离人工确认的事实承诺**
   - 事实、数字、时点、政策依据等关键内容必须可追溯、可复核
5. **以花哨交互替代底层能力建设**
   - UI 可后置，先把制度、语料、规则和工作流打牢

## 3.3 红线边界

以下为必须严格执行的项目红线：

- 不得伪造政策依据、会议结论、领导指示、统计数据
- 不得把推测性内容伪装成既定事实
- 不得输出无法说明来源的重要表述
- 不得绕过人工审定直接形成对外正式发文行为
- 不得让 agent 擅自修改已确认口径且不留痕

## 4. 目标用户与典型场景

## 4.1 核心用户

- 内部综合办/办公室/秘书线
- 业务条线负责人及其材料协作人员
- 需要快速形成汇报材料与方案文本的项目负责人
- 主代理与三省六部执行体系中的各责任单元

## 4.2 典型场景

### 场景 A：从任务描述到首稿

输入：领导要求、会议任务、主题、对象、篇幅、截止时间  
输出：符合文种结构的首稿/提纲草稿

### 场景 B：从材料堆到可用文本

输入：多份零散材料、纪要、旧稿、政策摘录  
输出：结构化提纲、统一口径稿、重点要点抽取结果

### 场景 C：从草稿到审校版

输入：已有草稿  
输出：格式修订建议、逻辑缺口提示、风险表达标记、定稿清单

### 场景 D：从一次任务到长期资产

输入：完成过的优质稿件与审校记录  
输出：模板、句式、案例、规则、反例、可复用 SOP

## 5. 阶段划分

项目按“先跑通闭环，再打磨质量，再做体系扩张”的逻辑推进。

## 5.1 P0：项目立项与执行底座期

### 目标

把项目的最小闭环、角色边界、文档规范、资产目录先搭起来。

### 关键产出

- 项目总方案（本文）
- 目录结构与命名规范
- 角色分工说明
- 首版任务流转 SOP
- 首版文种清单与优先级
- 语料目录框架
- 验收口径初版

### 完成标志

- 主代理与三省六部知道各自做什么
- 新任务来了知道放哪、怎么拆、谁先动
- 不再依赖临场口头协调

## 5.2 P1：最小可用公文生产闭环

### 目标

聚焦少数高频文种，做出“可稳定出草稿”的最小版本。

### 重点文种建议

- 通知
- 请示
- 工作方案
- 汇报材料
- 会议纪要
- 工作总结

### 核心能力

- 文种识别
- 基于场景的提纲生成
- 模板驱动首稿生成
- 基础改写（压缩、扩写、重组）
- 基础审校（错别字、结构缺项、常见公文表达问题）

### 完成标志

- 高频文种可稳定出首稿
- 输出结构基本符合预期
- 可以支撑真实小规模内部试运行

## 5.3 P2：质量强化与审校治理期

### 目标

把“能写”提升为“写得稳、审得住、改得动”。

### 重点能力

- 事实/来源标注机制
- 风格与口径控制
- 风险表达检测
- 审校建议分级（必须改 / 建议改 / 可选优化）
- 多版本 diff 与反馈吸收
- 优秀稿件反向沉淀

### 完成标志

- 关键问题可自动提示
- 多轮反馈不会导致文本失控
- 审校环节可形成标准化操作

## 5.4 P3：知识化与多 agent 协同时期

### 目标

将单次任务能力提升为组织能力，让系统具备“越做越强”的积累性。

### 重点能力

- 语料分级治理
- 场景模板自动匹配
- 规则引擎与案例库联动
- agent 分工协同、留痕与责任归属
- 横向扩展到更多文种、部门和场景

### 完成标志

- 不同 agent 在同一制度下可稳定协作
- 优质经验会沉淀，而不是做完即散
- 项目具备持续迭代能力

## 6. 能力地图

以下能力地图按“输入理解 → 内容生成 → 质量控制 → 组织沉淀”四层组织。

## 6.1 输入理解层

负责把模糊任务转成结构化可执行输入。

### 核心能力

- 文种识别
- 任务要素抽取（主题、对象、场景、目标、口径、篇幅、时限）
- 材料归并与去重
- 历史稿件召回
- 政策/制度/事实材料抽取
- 缺失信息诊断

### 输出形式

- 任务卡
- 要素清单
- 材料包
- 缺口提示

## 6.2 内容生成层

负责从任务卡与材料包生成可编辑草稿。

### 核心能力

- 提纲规划
- 模板匹配
- 章节级写作
- 基于语料的表述复用
- 改写、续写、压缩、扩写、重组
- 多版本候选生成

### 输出形式

- 提纲版
- 首稿版
- 指定文风/文种重写版
- 篇幅裁剪版

## 6.3 质量控制层

负责把草稿变成可以进入内部审核流程的文本。

### 核心能力

- 格式与结构校验
- 逻辑链检查
- 敏感表达与风险提示
- 事实/数据/引用待核项标记
- 口径一致性检测
- 套话冗余识别
- 定稿前清单核对

### 输出形式

- 审校意见单
- 风险标记稿
- 修改建议清单
- 待人工确认事项

## 6.4 知识沉淀层

负责把一次性劳动转为长期资产。

### 核心能力

- 优质稿件入库
- 典型表述归档
- 模板抽象
- 反例与失败案例归档
- 审校规则更新
- 任务复盘与 SOP 优化

### 输出形式

- 模板库
- 语料库
- 规则库
- 案例库
- 复盘记录

## 6.5 能力优先级

### 第一优先级（必须先做）

- 文种识别
- 提纲生成
- 模板驱动首稿
- 基础审校
- 任务卡结构化

### 第二优先级（尽快补齐）

- 语料检索与表述复用
- 口径一致性控制
- 风险提示
- 多版本 diff
- 优秀稿件沉淀

### 第三优先级（体系成熟后做）

- 跨部门知识图谱
- 自动策略路由
- 更细颗粒度的风格控制
- 数据/事实自动核验联动

## 7. 语料策略

语料是这个项目的地基。没有语料治理，模型只会“写得像”，不会“写得准”。

## 7.1 语料总体原则

1. **先分层，再使用**：不是所有材料都能直接喂给生成链路。
2. **先可信，再丰富**：宁可少一点，也不要把低质量语料混进核心库。
3. **先可追溯，再求自动化**：重要内容必须知道从哪来。
4. **先案例化，再抽象化**：先积累真实稿件与修改记录，再归纳模板和规则。

## 7.2 语料分层

建议将语料至少分为五层：

### A 层：权威依据层

包括：

- 已公开政策文件
- 规范性文件
- 制度文件
- 权威会议纪要/批准口径
- 正式发布的通知、通报、决定等

用途：作为事实、口径、结构依据的最高优先级来源。

### B 层：内部定稿层

包括：

- 已经正式采用的历史公文
- 已通过审核的讲话稿、方案、总结、汇报稿
- 明确可复用的部门模板

用途：作为文风、结构、表述习惯的重要参考。

### C 层：工作材料层

包括：

- 会议记录
- 业务素材
- 汇总表格
- 线索材料
- 部门上报文字

用途：提供事实与内容素材，但默认不直接视为最终口径。

### D 层：生成中间层

包括：

- agent 草稿
- 提纲草案
- 多轮修改中间稿
- 审校意见稿

用途：用于过程留痕、版本对比、经验回收，不直接进入高可信复用库。

### E 层：规则与案例层

包括：

- 模板
- 句式规范
- 审校规则
- 常见错误案例
- 优秀改写对照样本

用途：支撑生成、审校和训练 SOP。

## 7.3 语料使用原则

- A/B 层可进入核心召回链路，但要标记来源与适用范围。
- C 层只用于事实补充与素材抽取，不得未经判断直接提升为正式口径。
- D 层只做过程资产，不得反向污染核心模板库。
- E 层作为制度资产，需要版本号和维护责任人。

## 7.4 语料治理动作

### 入库前

- 来源登记
- 文种标注
- 场景标注
- 时间标注
- 状态标注（权威 / 已定稿 / 参考 / 待核）
- 敏感级别标注

### 入库后

- 去重与相似稿归并
- 优秀表述抽取
- 常见段落模板抽取
- 无效套话清理
- 过时内容下线或降级

### 使用中

- 引用来源留痕
- 命中记录
- 人工反馈回收
- 错误追因到具体语料/规则/环节

## 7.5 语料演进路线

- 初期：人工精选少量高质量 A/B 层语料，确保可控
- 中期：逐步扩充 C 层素材抽取与 E 层规则资产
- 后期：建立动态评分机制，对语料质量、命中有效性、复用价值进行打分

## 8. 多 agent 制度要求

本项目不是“一个万能 agent 干到底”，而是“主代理统筹 + 多执行单元协同 + 关键节点人工把关”的制度。

## 8.1 角色原则

### 主代理

职责：

- 接收项目目标与任务
- 拆解子任务
- 路由到合适执行单元
- 汇总结果并做最终整合
- 维护总方案、总规则、总边界

主代理不应长期陷入具体微操作，否则会失去总控能力。

### 三省

可理解为三类统筹中枢：

1. **中书省（起草中枢）**
   - 负责文种理解、提纲设计、首稿组织、改写统筹
2. **门下省（审议中枢）**
   - 负责审校、风控、逻辑一致性、口径把关
3. **尚书省（执行中枢）**
   - 负责任务分发、材料管理、流程推进、版本留痕、归档沉淀

### 六部

建议映射为六类专业执行能力，可按项目需要调整命名，但职能要稳定：

1. **吏部：任务登记与调度**
   - 建任务卡、分优先级、控节点、追时限
2. **户部：语料与资产管理**
   - 管模板、语料、案例、元数据、版本状态
3. **礼部：文风与格式规范**
   - 管文种体例、礼貌边界、格式细则、表达习惯
4. **兵部：风险与应急审校**
   - 管敏感表达、对外风险、冲突口径、时效场景
5. **刑部：规则稽核与责任留痕**
   - 管红线校验、审校清单、问题归因、责任记录
6. **工部：能力工程与工具链**
   - 管提示词、流程编排、检索链路、评测脚本、系统优化

## 8.2 任务流转规则

统一采用以下最小流转链：

1. 任务进入 → 吏部登记任务卡
2. 材料收集 → 户部整理材料与语料包
3. 起草组织 → 中书省生成提纲/首稿
4. 规范修整 → 礼部校正文种和格式
5. 风险审议 → 门下省 + 兵部做风险与逻辑审校
6. 稽核留痕 → 刑部记录问题、责任和修改意见
7. 能力回收 → 尚书省归档，工部沉淀规则和流程改进

## 8.3 制度要求

### 要求 1：任何 agent 都不能“黑箱定稿”

必须保留：

- 输入材料
- 关键决策依据
- 输出版本
- 审校意见
- 关键修改原因

### 要求 2：起草与审校必须分离

同一 agent 可参与多个环节，但不能既当运动员又当裁判而不留第二视角。关键稿件必须至少经过一次独立审校视角。

### 要求 3：意见必须分级

审校意见至少分为：

- 阻断项：不改不能流转
- 高优项：建议本轮处理
- 一般项：可视时间优化
- 观察项：记录但不强制修改

### 要求 4：口径变更必须显式记录

凡涉及：

- 立场变化
- 结论变化
- 数据变化
- 对外表述变化
- 政策依据变化

必须记录“谁改的、为什么改、依据是什么”。

### 要求 5：复用资产必须过筛

不是所有完成稿都能进模板库。只有通过审核、复用价值高、时效仍有效的材料才允许升格为可复用资产。

## 8.4 协同纪律

- 先产出，再讨论，不允许长期停留在空泛计划层。
- 有不确定项要显式打标，不允许把猜测混写成结论。
- 不重复造轮子，先查模板库、规则库、案例库。
- 不为追求“像人写的”而牺牲明确性、准确性、可审性。
- 不允许跨边界擅自删除关键留痕。

---

## 9. 验收原则

验收不看单次演示效果，而看真实工作流中的稳定性、可控性和复用价值。

## 9.1 总体验收原则

1. **先验闭环，不先验炫技**
   - 是否能完整跑通任务到归档
2. **先验稳定，不先验偶然高光**
   - 不能只看一篇写得好
3. **先验可控，不先验自动化程度**
   - 可控比全自动更重要
4. **先验可追责，不先验表面流畅**
   - 出问题能追到环节，才算可用

## 9.2 交付物验收维度

### A. 任务理解准确性

- 是否正确识别文种
- 是否抓住任务目标与对象
- 是否发现关键信息缺口
- 是否能组织成结构化任务卡

### B. 内容生成质量

- 结构是否符合文种常规
- 内容是否围绕任务目标展开
- 是否存在明显套话堆积
- 是否能在指定篇幅下保持完整性

### C. 审校有效性

- 是否能发现结构缺项
- 是否能标出风险表达
- 是否能指出事实待核项
- 审校意见是否可执行，而非空泛评价

### D. 流程可控性

- 是否保留版本与意见留痕
- 是否能区分草稿、审校稿、定稿
- 是否能记录口径变更
- 是否能回收经验到规则/模板/案例库

### E. 组织复用价值

- 新任务是否复用了旧资产
- 旧资产是否因本次任务得到强化
- 是否形成新的模板/规则/案例

## 9.3 验收分级建议

### 不通过

- 文种判断错误
- 结构严重失真
- 存在明显事实编造或口径错误
- 无法解释关键内容来源
- 无版本留痕

### 通过但需整改

- 基本能用，但审校问题较多
- 模板复用不稳定
- 多轮改写后质量波动较大
- 知识沉淀动作不足

### 良好

- 高频场景可以稳定出可编辑稿
- 审校链路有效
- 有明确语料和规则沉淀
- 多 agent 协作顺畅

### 优秀

- 能支撑真实业务连续使用
- 质量波动小
- 问题定位快
- 复用效率明显提升
- 项目具备“越用越强”的组织效应

## 10. 执行建议与近期动作

## 10.1 近期优先事项

建议按以下顺序推进：

1. 明确首批高频文种范围
2. 建立任务卡字段标准
3. 建立语料目录和元数据规范
4. 沉淀首版模板库与审校清单
5. 设计最小流转 SOP
6. 选取一批真实样本做小规模试跑
7. 基于试跑结果更新规则、模板和分工边界

## 10.2 首批必须形成的内部资产

- 《文种清单与优先级》
- 《任务卡字段规范》
- 《语料入库规范》
- 《首版模板库》
- 《首版审校规则清单》
- 《版本命名与留痕规范》
- 《试运行验收样本集》

## 10.3 管理口径

对内部执行单元统一要求：

- 先做可用版本，再做理想版本
- 先把高频场景打穿，再扩展低频长尾
- 先保证事实和结构正确，再优化文采和修辞
- 任何结论都要能回到依据、语料和规则

这份《项目总方案》不是展示稿，而是执行母版，后续用途包括：

- 作为三省六部划分职责的制度底稿

## 12. 一句话结论

公文助手项目的本质，是把“公文写作能力”从一次次临时发挥，升级为**有边界、有制度、有语料、有审校、有沉淀的内部协同生产系统**。

先把闭环做实，再把质量做稳，最后把组织能力做出来。
