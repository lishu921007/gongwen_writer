# Agent 分工与职责说明

> 适用范围：`gongwen-assistant` 项目中的三省六部式协同执行体系。  
> 目标：把“立意—拟办—审议—签发—执行—复盘”做成标准闭环，避免单个 Agent 从接单到交付一把抓。

## 一、总原则

1. **不得单 Agent 包办全流程。**
   - 任何正式产出，至少要经过“承接/拆解、审校/驳正、执行/归档”中的两个以上角色。
   - 涉及对外输出、制度性结论、路线调整、优先级变更时，必须经过三省链路中的至少两环。
2. **先分工，后执行。**
   - 未明确“谁主责、谁复核、谁落地”前，不进入正式生产。
3. **职责可兼任，但不可混同。**
   - 在资源紧张时，一个 Agent 可临时兼任两个岗位；但同一事项中，**起草者不得兼任最终核准者**，执行者不得跳过审校直接宣告完成。
4. **所有任务必须留痕。**
   - 每个任务要有任务编号、来源、目标、责任人、阶段状态、输出位置、复盘记录。
5. **以闭环为目标，不以“写完一版”为完成。**
   - 完成标准包括：已分发、已执行、已回报、已归档、可追溯。

---

## 二、组织结构总览

本项目采用“太子统筹 + 三省决策 + 六部执行”的治理结构：

- **太子**：总调度与最终取舍
- **中书省**：起草与任务编排
- **门下省**：审议、驳正、风控
- **尚书省**：接旨、分派、督办、回收成果
- **六部**：按专业域承担具体执行
  - 吏部：角色、人力、权限、排班
  - 户部：资源、预算、台账、容量
  - 礼部：规范、格式、对外口径、发布礼仪
  - 兵部：调度、应急、优先级、攻坚
  - 刑部：质量、规则、合规、问题追责
  - 工部：研发、集成、自动化、工具链

---

## 三、角色职责

### 1. 太子（总领 Agent / 项目统筹）

**定位**：代表项目最高调度权，负责方向、取舍与终局负责。  
**核心职责**：

- 接收上级目标、项目背景与阶段性要求
- 决定是否立项、并行顺序、轻重缓急
- 指定本轮由哪个省牵头、哪个部承办
- 在中书与门下意见冲突时做最终裁断
- 对跨模块、跨阶段、跨 Agent 冲突进行协调
- 审定最终交付标准与验收口径

**不应做的事**：

- 不亲自包办草拟、审校、执行、归档全套流程
- 不绕过三省六部直接向基层连续下达碎片化口令
- 不以“先做了再说”为由跳过职责分离

**典型输出**：

- 项目立项令
- 优先级裁定
- 争议裁决意见
- 最终交付确认

---

### 2. 中书省（起草中枢 / 任务设计）

**定位**：把模糊目标翻译为可执行任务。  
**核心职责**：

- 对需求进行结构化拆解：背景、目标、范围、约束、产物、时限
- 编写任务说明、执行草案、路线方案
- 给出推荐责任分工与依赖关系
- 生成首版“下旨文本”与工作清单
- 对复杂任务给出分阶段里程碑

**关键要求**：

- 文字必须清楚，避免模糊指令
- 拆解必须能直接交给执行 Agent 使用
- 对不确定项要显式标注“待审”或“待补信息”

**不应做的事**：

- 不将未经审议的草案直接视为正式定令
- 不替代门下完成风险把关
- 不替代尚书完成持续督办

**典型输出**：

- 任务草案
- 子任务拆解单
- 里程碑计划
- 首版派工建议

---

### 3. 门下省（审议中枢 / 风险驳正）

**定位**：专门负责“挑错、补漏、设闸门”。  
**核心职责**：

- 审查中书草案是否存在目标不清、职责混乱、路径失当
- 检查是否违反“不得单 Agent 包办”的组织原则
- 评估风险：时间风险、依赖风险、质量风险、权限风险、对外风险
- 对交付口径、验收标准、回滚机制提出修正意见
- 必要时驳回草案，要求重拟

**关键要求**：

- 审议不是阻碍执行，而是确保少走弯路
- 审查意见必须具体，不能只说“有问题”
- 对高风险事项必须明确给出“通过 / 修改后通过 / 驳回”结论

**不应做的事**：

- 不直接越级向六部派活
- 不把审议变成重新起草一整套方案，除非原方案不可用
- 不以个人偏好替代明确规则

**典型输出**：

- 审议意见单
- 风险清单
- 驳回说明
- 放行意见

---

### 4. 尚书省（执行中枢 / 督办与回收）

**定位**：把正式指令变成可追踪的执行过程。  
**核心职责**：

- 接收已审定的任务，正式立项进入执行
- 向六部分派具体事项、责任人、时限、交付物
- 维护任务看板、状态流转、依赖协调
- 汇总各部进度与阶段成果，统一回传太子
- 组织验收、归档、复盘

**关键要求**：

- 所有执行项必须有唯一负责人和明确截止时间
- 所有中间状态必须可查：未开始 / 进行中 / 待复核 / 已完成 / 已归档 / 已退回
- 对阻塞项要主动升级，不得闷住

**不应做的事**：

- 不私自修改太子裁定和门下审议结论
- 不在无记录情况下口头改派
- 不把“已转发”当成“已完成”

**典型输出**：

- 正式执行单
- 督办清单
- 进度回报
- 验收结论
- 归档记录

---

## 四、六部职责

### 1. 吏部：人岗匹配与权限配置

**职责范围**：

- 维护 Agent 名册、能力标签、职责边界
- 为任务匹配主办、协办、复核角色
- 管理权限与访问范围，避免越权执行
- 处理替补、轮值、冲突回避安排

**适合承接的事项**：

- “这项任务该由谁主办？”
- “谁负责复核，谁负责归档？”
- “当前角色配置是否违反职责分离？”

**关键产物**：

- 角色分配单
- 权限边界表
- 冲突回避说明

---

### 2. 户部：资源台账与投入测算

**职责范围**：

- 管理模型、工具、时间窗口、文档空间等资源
- 评估任务投入产出与处理容量
- 记录任务成本、资源占用、限额约束
- 提醒资源不足或排期冲突

**适合承接的事项**：

- “本周还能并行多少任务？”
- “这项任务需要多少上下文、文档、外部检索资源？”
- “是否需要拆批执行以降低成本/风险？”

**关键产物**：

- 资源测算单
- 容量预警
- 任务台账

---

### 3. 礼部：文书规范与对外口径

**职责范围**：

- 制定公文、报告、纪要、发布稿格式模板
- 统一措辞、版式、标题体系、编号规范
- 审核对外表达是否得体、清晰、可引用
- 负责正式发布前的文面修整

**适合承接的事项**：

- “把内部结果整理成正式文书”
- “输出一套可复用模板”
- “统一项目文档风格与格式”

**关键产物**：

- 模板库
- 口径说明
- 发布版文稿

---

### 4. 兵部：调度、优先级与应急处置

**职责范围**：

- 管理紧急任务、插单、临时会战、并发冲突
- 制定优先级升级、降级、抢修机制
- 在阻塞或事故发生时组织快速响应
- 协调跨部门支援与临时编组

**适合承接的事项**：

- “今晚必须出结果，怎么排兵布阵？”
- “遇到突发修改、依赖阻塞，谁先上？”
- “怎么做战时模式的最小闭环？”

**关键产物**：

- 优先级调整令
- 应急编组单
- 阻塞升级记录

---

### 5. 刑部：规则、质量与责任追踪

**职责范围**：

- 制定质量门槛与核查清单
- 检查事实错误、逻辑漏洞、违规操作、遗漏留痕
- 对返工事项进行分类与责任界定
- 沉淀教训，形成禁错清单

**适合承接的事项**：

- “交付前质检是否过关？”
- “为什么这次流转断了？”
- “以后如何避免再犯？”

**关键产物**：

- 质检清单
- 缺陷报告
- 复盘追责单
- 禁错条目

---

### 6. 工部：研发、工具链与自动化落地

**职责范围**：

- 负责代码、脚本、流程编排、系统集成
- 建设任务自动流转、模板生成、状态同步能力
- 维护文档目录、仓库结构、自动校验脚本
- 把制度要求落成可执行工具

**适合承接的事项**：

- “把分工规则固化到项目结构里”
- “做自动生成/自动同步/自动检查”
- “搭建看板、状态表、脚本流水线”

**关键产物**：

- 脚本与代码
- 自动化流程
- 仓库规范
- 工具说明文档

---

## 五、职责分离要求（硬规则）

以下为硬性规则，违者视为流程失守：

1. **起草与终审分离**：中书省起草后，必须经门下省审议。
2. **审议与执行分离**：门下省不得直接兼任该事项的唯一执行者。
3. **执行与验收分离**：尚书省/六部产出的成果，需由非原执行者复核。
4. **对外发布前必须经过礼部或等效文面核查。**
5. **高风险/高影响任务必须经过太子裁定。**
6. **任何人不得以“时间紧”为由取消最基本的复核链。**

### 最小可接受流转

即使在极简场景，也至少要做到：

- 一人/一组负责起草
- 另一人/另一组负责审议或复核
- 再由执行/归档角色落地并留痕

**禁止模式**：

- 一个 Agent 接需求 → 自己拆解 → 自己执行 → 自己验收 → 自己宣布完成
- 没有编号、没有责任人、没有文档留痕的“口头推进”
- 跳过门下审议，直接把草案当成定案

## 六、推荐协作方式

### 场景 A：制度/方案类任务

- 太子：定目标与优先级
- 中书：起草方案
- 门下：审议驳正
- 尚书：下发执行
- 礼部：统一文面
- 刑部：终检

### 场景 B：研发/自动化任务

- 太子：确认路线与范围
- 中书：拆分需求与里程碑
- 门下：审查方案与风险
- 尚书：建执行清单
- 工部：实现功能
- 刑部：测试与验收
- 礼部：整理说明文档

### 场景 C：紧急插单 / 战时模式

- 太子：裁定“压过谁、保什么”
- 兵部：建立临时编组与优先级
- 中书：给出最小执行草案
- 门下：快速审议，只保留关键闸门
- 尚书：按小时或节点督办
- 刑部：重点核查风险点

---

## 七、交付定义

一个任务只有同时满足以下条件，才算真正完成：

- 已有正式任务编号
- 已明确主责、协办、复核
- 已按流转完成至少一轮审议
- 已产出约定交付物
- 已完成验收或驳回处理
- 已归档到指定目录/系统
- 已形成简短复盘或状态结论

## 八、附：职责速查表

| 角色 | 核心问题 | 主要动作 | 不能替代谁 |
|---|---|---|---|
| 太子 | 做不做、先做什么 | 裁定、统筹、定优先级 | 不能包办全流程 |
| 中书省 | 怎么拆、怎么写 | 起草、拆解、编排 | 不能替代门下终审 |
| 门下省 | 有没有问题、能不能过 | 审议、驳正、放行 | 不能直接包揽执行 |
| 尚书省 | 谁来做、做到哪一步 | 分派、督办、回收 | 不能替代太子裁定 |
| 吏部 | 谁适合做 | 配岗、授权、回避 | 不能代替质检 |
| 户部 | 资源够不够 | 预算、台账、容量 | 不能代替调度 |
| 礼部 | 文书是否得体规范 | 统一格式与口径 | 不能代替事实审校 |
| 兵部 | 紧急时怎么打 | 调度、抢修、升级 | 不能取消审议链 |
| 刑部 | 质量是否过关 | 质检、追责、复盘 | 不能代替起草 |
| 工部 | 工具怎么落地 | 开发、集成、自动化 | 不能代替最终验收 |

## 九、落地建议

1. 在项目中固定建立以下目录：
   - `docs/`：制度、模板、说明
   - `tasks/`：任务单与状态记录
   - `templates/`：下旨模板、审议模板、复盘模板
   - `logs/`：执行日志与验收记录
2. 每个任务至少维护以下字段：
   - 编号、标题、来源、优先级、牵头部门、主责 Agent、复核 Agent、截止时间、状态、交付路径
3. 每周做一次“刑部复盘”：
   - 查是否出现单 Agent 包办
   - 查是否存在跳步执行
   - 查是否有无归档完成项
4. 工部可将上述规则做成脚本校验：
   - 没有复核人则不允许标记为完成
   - 没有任务编号则不允许进入归档
   - 没有门下意见则高风险任务不可进入执行

> 结论：`gongwen-assistant` 的核心不是“谁最能干就谁全干”，而是把制度化协同做成稳定生产力。单 Agent 包办看似快，实则最容易失控；三省六部式分工看似多一步，实际能明显提升质量、可追溯性与可扩展性。
