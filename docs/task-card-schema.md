# 《任务卡字段规范》

> 文档属性：内部执行规范（首版）  
> 适用范围：`gongwen-assistant` 项目中，由三省六部承接、流转、起草、审校、归档的各类公文任务卡  
> 文档目标：统一任务卡字段口径，解决“接单描述模糊、分发标准不一、关键信息缺失、后续难以追责与复用”的问题。

---

## 1. 为什么要有任务卡字段规范

公文项目的难点，不只是“写一篇稿子”，而是把一个模糊任务稳定变成：

1. 可分发的执行单
2. 可起草的输入包
3. 可审校的判断对象
4. 可归档的过程资产
5. 可复用的知识样本

如果任务卡字段不统一，就会出现：

- 有人只说“写个汇报”，没人知道是汇报给谁
- 有人知道要写“通知”，却不知道是否正式发文
- 有人起草了稿子，却不知道事实材料从哪里来
- 有人改了口径，却没有记录谁改的、为何改
- 有人做完了，但后续沉淀不到模板库和案例库

因此，任务卡必须成为整个项目的**标准入口对象**。

一句话：**任务卡不是备注单，而是公文任务的结构化母表。**

---

## 2. 使用原则

## 2.1 三类字段状态

本规范将任务卡字段分为三类：

### A. 必填字段

- 接单时原则上必须补齐
- 若暂缺，必须显式标记“待补”或“待确认”
- 缺失会影响起草、审校、派工或验收

### B. 选填字段

- 对部分场景很重要，但不是所有任务都必须有
- 没有时可以为空，不阻断流转
- 一旦有，就应尽量结构化填写，而不是塞进自由文本

### C. 自动生成字段

- 由系统、流程或执行环节自动写入
- 原则上不依赖人工手填
- 用于留痕、路由、统计、归档和复盘

## 2.2 填写原则

1. **先保证可执行，再追求完整细腻。**
2. **不确定的内容要打标，不要伪装成已确认。**
3. **自由文本不是兜底垃圾桶，能结构化就结构化。**
4. **事实、口径、风险相关字段优先级高于文采类字段。**
5. **涉及正式发文、领导口径、政策依据、数据事实的字段，必须可追溯。**

## 2.3 一张任务卡服务谁

同一张任务卡，至少服务以下角色：

- **吏部**：据此登记、分级、派工
- **户部**：据此组织材料包和历史语料
- **中书省**：据此设计提纲、生成首稿
- **礼部**：据此检查文种体例与文面要求
- **门下省 / 兵部 / 刑部**：据此识别风险、审校、验收
- **尚书省 / 工部**：据此流转、留痕、沉淀为结构化资产

---

## 3. 任务卡总体结构

建议将任务卡拆为九个字段组：

1. 基本登记信息
2. 公文任务核心信息
3. 写作目标与输出要求
4. 材料与事实来源
5. 风格与口径控制
6. 流转与审校控制
7. 时限与优先级控制
8. 交付与归档信息
9. 系统自动留痕信息

---

## 4. 字段清单（按字段组展开）

以下为首版推荐字段。

---

## 4.1 基本登记信息

### 4.1.1 任务编号

- 字段名：`task_id`
- 类型：字符串
- 状态：**自动生成**
- 说明：任务唯一编号，用于全流程追踪
- 示例：`GW-2026-03-030`

### 4.1.2 任务标题

- 字段名：`task_title`
- 类型：字符串
- 状态：**必填**
- 说明：用一句话概括任务，不要只写“写材料”“改稿子”
- 示例：`起草关于推进基层减负专项整治情况的专题汇报材料`

### 4.1.3 任务来源

- 字段名：`task_source`
- 类型：字符串 / 枚举
- 状态：**必填**
- 说明：写清任务来自领导指示、会议安排、上级来文、部门申请、临时交办等
- 示例：`厅领导临时交办`

### 4.1.4 发起单位/需求提出方

- 字段名：`requesting_unit`
- 类型：字符串
- 状态：**必填**
- 说明：谁提出了这项任务
- 示例：`办公室综合处`

### 4.1.5 牵头单位

- 字段名：`owner_unit`
- 类型：字符串
- 状态：**选填**（立项后建议补齐）
- 说明：最终由哪个部门牵头推进
- 示例：`政策法规处`

### 4.1.6 当前状态

- 字段名：`task_status`
- 类型：枚举
- 状态：**自动生成**
- 说明：如待立项、待起草、待审议、进行中、待复核、已完成、已归档
- 示例：`待起草`

---

## 4.2 公文任务核心信息

这一组是公文项目最关键的字段，必须重点规范。

### 4.2.1 文种

- 字段名：`document_type`
- 类型：枚举
- 状态：**必填**
- 说明：必须明确属于哪类公文或材料，不能只写“文章”“文稿”
- 推荐值：
  - 通知
  - 请示
  - 报告
  - 工作方案
  - 领导讲话稿
  - 汇报材料
  - 会议纪要
  - 函
  - 通报
  - 总结
  - 简报
  - 提纲
  - 其他（需补充说明）
- 示例：`汇报材料`

### 4.2.2 公文对象/报送对象

- 字段名：`target_audience`
- 类型：字符串 / 数组
- 状态：**必填**
- 说明：写清给谁看、报给谁、发给谁。对象不同，结构和措辞差异很大。
- 示例：`报送省政府办公厅`、`面向市县各部门印发`

### 4.2.3 主送对象级别

- 字段名：`target_level`
- 类型：枚举
- 状态：**选填**
- 说明：用于辅助判断语气和规格
- 推荐值：`中央`、`省级`、`厅局级`、`市县级`、`内部会议`、`对下通知`
- 示例：`省级`

### 4.2.4 主题事项

- 字段名：`topic`
- 类型：字符串
- 状态：**必填**
- 说明：本次任务围绕什么事项，不要与标题完全重复，可更聚焦业务内容
- 示例：`基层减负专项整治进展及下一步安排`

### 4.2.5 使用场景

- 字段名：`scenario`
- 类型：枚举 / 字符串
- 状态：**必填**
- 说明：明确此稿用于什么场景
- 推荐值：
  - 上会汇报
  - 对上请示
  - 对下部署
  - 内部通报
  - 迎检迎查
  - 领导讲话
  - 专项调度
  - 外部协调
  - 存档备查
  - 其他
- 示例：`上会汇报`

### 4.2.6 任务类型

- 字段名：`task_type`
- 类型：枚举
- 状态：**必填**
- 说明：是从零起草，还是改写、压缩、扩写、审校、重组
- 推荐值：`新起草`、`基于旧稿改写`、`扩写`、`压缩`、`重组`、`审校修订`、`提纲生成`
- 示例：`基于旧稿改写`

### 4.2.7 规格/篇幅要求

- 字段名：`specification`
- 类型：对象 / 字符串
- 状态：**必填**
- 说明：建议拆成结构化子项，而不是只写“控制一下篇幅”
- 推荐子项：
  - `length_hint`：字数范围
  - `structure_hint`：是否分标题、是否三级标题
  - `format_hint`：是否要正式公文格式、汇报提纲格式、PPT 讲稿式等
- 示例：
  - `字数控制在 2500-3000 字`
  - `分三部分，每部分 2-3 个小点`
  - `采用正式汇报材料结构`

### 4.2.8 是否正式发文

- 字段名：`is_formal_dispatch`
- 类型：布尔值
- 状态：**选填**
- 说明：判断是否按正式公文要求处理
- 示例：`false`

---

## 4.3 写作目标与输出要求

### 4.3.1 写作目标

- 字段名：`writing_goal`
- 类型：字符串
- 状态：**必填**
- 说明：这篇稿子要达成什么，不只是“写出来”，而是“说服、汇报、争取、部署、解释、定调”哪一种
- 示例：`向省政府汇报专项整治成效，同时争取后续试点政策支持`

### 4.3.2 关键结论/核心观点

- 字段名：`key_points`
- 类型：数组
- 状态：**选填**
- 说明：希望稿件最终必须表达出的中心判断
- 示例：
  - `专项整治已取得阶段性成效`
  - `基层反映较强的问题已纳入整改`
  - `仍需省级层面统筹机制支持`

### 4.3.3 输出形式

- 字段名：`output_format`
- 类型：枚举 / 数组
- 状态：**必填**
- 说明：明确最终交付是什么形态
- 推荐值：
  - 完整首稿
  - 提纲
  - 审校意见单
  - 修订稿
  - 对比稿
  - 摘要版
  - 讲话提词版
  - 表格化要点
- 示例：`完整首稿 + 300字摘要版`

### 4.3.4 是否需要多版本输出

- 字段名：`need_multi_versions`
- 类型：布尔值
- 状态：**选填**
- 说明：是否同时产出长版/短版、正式版/口语版等
- 示例：`true`

### 4.3.5 版本要求说明

- 字段名：`version_requirements`
- 类型：字符串
- 状态：**选填**
- 说明：多版本场景下的具体要求
- 示例：`需同时提供 3000 字会签版与 800 字领导口头汇报版`

---

## 4.4 材料与事实来源

这一组直接关系到真实性和可追溯性，优先级非常高。

### 4.4.1 事实材料来源

- 字段名：`fact_sources`
- 类型：数组
- 状态：**必填**
- 说明：必须写清稿件主要事实材料从哪里来
- 推荐来源类型：
  - 政策文件
  - 已定稿历史公文
  - 会议纪要
  - 部门报送材料
  - 统计报表
  - 领导批示
  - 调研记录
  - 新闻通稿
  - 待核材料
- 示例：
  - `2026年一季度专项整治统计台账`
  - `3月18日专题调度会纪要`
  - `各地市报送情况汇总表`

### 4.4.2 权威依据/政策依据

- 字段名：`authority_basis`
- 类型：数组
- 状态：**选填**（涉及政策口径时应视同必填）
- 说明：支撑正式表述的政策、制度、上级文件依据
- 示例：
  - `《关于持续整治形式主义为基层减负的若干规定》`
  - `省政府办公厅〔2026〕12号文`

### 4.4.3 历史参考稿

- 字段名：`reference_drafts`
- 类型：数组
- 状态：**选填**
- 说明：可参考但不可机械照抄的旧稿或模板
- 示例：`2025年同主题季度汇报稿（定稿版）`

### 4.4.4 数据是否需人工复核

- 字段名：`require_data_review`
- 类型：布尔值
- 状态：**必填**
- 说明：凡涉及数字、比例、同比环比、时间节点，默认建议判断是否需复核
- 示例：`true`

### 4.4.5 事实风险备注

- 字段名：`fact_risk_notes`
- 类型：字符串
- 状态：**选填**
- 说明：记录事实层面的不确定性
- 示例：`两地市报送口径尚未完全统一，涉案数量暂按初报口径写“约”`

---

## 4.5 风格与口径控制

### 4.5.1 风格目标

- 字段名：`style_goal`
- 类型：枚举 / 字符串
- 状态：**必填**
- 说明：不是抽象写“写好一点”，而要明确偏向
- 推荐值：
  - 庄重规范
  - 简洁有力
  - 稳妥克制
  - 汇报型
  - 部署型
  - 讲话型
  - 问题导向型
  - 成效总结型
- 示例：`稳妥克制 + 汇报型`

### 4.5.2 口径要求

- 字段名：`tone_constraints`
- 类型：字符串 / 数组
- 状态：**选填**
- 说明：明确哪些话能说、哪些话不能说，是否需与既有口径保持一致
- 示例：
  - `避免使用“全面解决”“彻底根治”等绝对化表述`
  - `与3月专题会议口径保持一致`

### 4.5.3 禁用表达

- 字段名：`forbidden_phrases`
- 类型：数组
- 状态：**选填**
- 说明：不允许出现的敏感或不合适表述
- 示例：
  - `一刀切`
  - `零问题`
  - `彻底完成`

### 4.5.4 指定参考风格样本

- 字段名：`style_references`
- 类型：数组
- 状态：**选填**
- 说明：指定接近的历史稿件、领导常用风格或部门范式
- 示例：`参考2025年全省专项整治推进会汇报稿的写法`

### 4.5.5 是否需升格

- 字段名：`need_escalation`
- 类型：布尔值
- 状态：**必填**
- 说明：是否需要把原始业务材料“升格”为更正式、更高站位、更适配上级机关的表达
- 示例：`true`

### 4.5.6 升格方向

- 字段名：`escalation_direction`
- 类型：字符串
- 状态：**选填**（`need_escalation=true` 时建议必填）
- 说明：升格不是空泛“拔高”，要写清往什么方向升
- 示例：
  - `从处室工作总结升格为省级专题汇报口径`
  - `从执行情况说明升格为领导讲话提纲`

## 4.6 流转与审校控制

### 4.6.1 是否需审稿

- 字段名：`need_review`
- 类型：布尔值
- 状态：**必填**
- 说明：是否必须进入独立审校环节。正式稿件原则上应为 true。
- 示例：`true`

### 4.6.2 审稿重点

- 字段名：`review_focus`
- 类型：数组
- 状态：**选填**
- 说明：本轮审稿重点看什么
- 推荐值：
  - 文种体例
  - 逻辑结构
  - 事实准确
  - 数据核对
  - 风险表达
  - 政策口径
  - 格式规范
  - 套话冗余
- 示例：`["事实准确", "政策口径", "风险表达"]`

### 4.6.3 风险等级

- 字段名：`risk_level`
- 类型：枚举
- 状态：**选填**
- 说明：便于兵部、门下省、刑部判断是否加严把关
- 推荐值：`低`、`中`、`高`、`特别高`
- 示例：`高`

### 4.6.4 是否涉领导口径

- 字段名：`involve_leadership_tone`
- 类型：布尔值
- 状态：**选填**
- 说明：若涉及领导讲话、批示转述、领导对外表态，风险显著提高
- 示例：`true`

### 4.6.5 是否允许模型补写缺失内容

- 字段名：`allow_reasoned_fill`
- 类型：布尔值
- 状态：**必填**
- 说明：决定系统在缺材料时是“先空着”还是“允许按合理框架补足非关键连接段”
- 示例：`false`

### 4.6.6 缺失信息处理策略

- 字段名：`missing_info_strategy`
- 类型：枚举 / 字符串
- 状态：**选填**
- 说明：建议固定策略，避免不同执行人理解不一
- 推荐值：
  - 缺失则打标待补
  - 可写框架不写事实
  - 允许补写过渡句但不得补事实
  - 退回补充材料
- 示例：`允许补写过渡句但不得补事实`

---

## 4.7 时限与优先级控制

### 4.7.1 优先级

- 字段名：`priority`
- 类型：枚举
- 状态：**必填**
- 说明：用于调度
- 推荐值：`紧急`、`高`、`中`、`低`
- 示例：`紧急`

### 4.7.2 截止时间

- 字段名：`deadline`
- 类型：日期时间
- 状态：**必填**
- 说明：最晚提交时间
- 示例：`2026-03-31 10:00`

### 4.7.3 是否有硬时点

- 字段名：`has_hard_deadline`
- 类型：布尔值
- 状态：**选填**
- 说明：是否存在会议、报送窗口、领导审签窗口等不可拖延节点
- 示例：`true`

### 4.7.4 硬时点说明

- 字段名：`deadline_reason`
- 类型：字符串
- 状态：**选填**
- 说明：解释为何必须在此时间前完成
- 示例：`下午14:30专题会前需完成领导审阅版`

---

## 4.8 交付与归档信息

### 4.8.1 交付路径

- 字段名：`delivery_path`
- 类型：字符串
- 状态：**自动生成 / 选填**
- 说明：文件、目录、系统链接等
- 示例：`projects/gongwen-assistant/tasks/GW-2026-03-030/`

### 4.8.2 期望沉淀类型

- 字段名：`asset_sink`
- 类型：数组
- 状态：**选填**
- 说明：任务完成后希望沉淀成什么资产
- 推荐值：`模板库`、`案例库`、`规则库`、`反例库`、`语料库`
- 示例：`["案例库", "规则库"]`

### 4.8.3 是否可复用

- 字段名：`reusable_after_review`
- 类型：布尔值
- 状态：**选填**
- 说明：该稿件或任务经验是否可能进入后续资产库

## 9. 字段设计上的几个硬要求

## 9.1 “文种 + 对象 + 场景”必须同时出现

只填其中一个不够。

- 只有文种，没有对象：不知道怎么措辞
- 只有对象，没有场景：不知道是请示还是汇报
- 只有场景，没有文种：容易写成四不像

所以这三个字段应视为任务理解层的核心三角。

## 9.2 “事实材料来源”不能省

公文项目最大的风险不是“写得不漂亮”，而是：

- 事实来源不明
- 口径依据不清
- 数据无法复核
- 推测被写成既定事实

因此 `fact_sources` 必须进入任务卡主表，不能只藏在附件里。

## 9.3 “是否需升格”必须独立成字段

很多公文任务不是从零写，而是把：

- 处室总结
- 基层材料
- 调研碎片
- 会议发言提要

升格为：

- 领导汇报稿
- 省级请示稿
- 正式部署通知
- 规范化讲话提纲

如果不单独写出 `need_escalation`，执行人会误把“整理材料”当作“照着改顺”。这会直接导致站位不够、语气不对、规格失真。

## 9.4 “是否需审稿”不能默认省略

不是所有任务都需要高强度审稿，但任何接近正式使用的稿件，原则上都要明确：

- 审不审
- 谁来审
- 重点审什么

否则流程上会默默滑向“起草即定稿”。

## 9.5 “是否允许补写缺失内容”必须有边界

这是 AI 场景中特别重要的一条。

同样是材料不全：

- 有的场景可以补写过渡句
- 有的场景可以先搭框架
- 有的场景绝不能补事实
- 有的场景必须退回补材料

如果没有 `allow_reasoned_fill` 和 `missing_info_strategy`，不同执行单元会各凭感觉处理，风险极高。

## 10. 落地建议

## 10.1 首期表单化建议

建议首期将字段分三层展示：

### 第一层：开单必填

只显示最小必填集合，保证任务能进系统。

### 第二层：高级补充

在需要时补：

- 权威依据
- 风格样本
- 禁用表达
- 升格方向
- 多版本要求
- 沉淀去向

### 第三层：系统留痕

自动生成，不让人工干预：

- 编号
- 状态
- 时间戳
- 版本号
- 角色
- 变更说明

## 10.2 后续结构化建议

后续若做 JSON Schema / 数据表，可按以下方向实施：

- 高频枚举字段强制下拉选择
- `fact_sources`、`authority_basis` 支持附件或链接挂接
- `specification` 设计为对象，而不是长文本
- `review_focus`、`asset_sink` 设计为多选
- 对 `need_escalation=true` 触发显示 `escalation_direction`
- 对 `need_review=true` 触发显示 `review_focus`
- 对 `has_hard_deadline=true` 触发显示 `deadline_reason`

## 10.3 与三省六部职责的映射建议

- **吏部**：维护 `task_id / priority / deadline / assignee_role`
- **户部**：维护 `fact_sources / authority_basis / reference_drafts`
- **中书省**：重点消费 `document_type / target_audience / scenario / writing_goal / style_goal`
- **礼部**：重点消费 `document_type / specification / style_goal / forbidden_phrases`
- **门下省 / 刑部**：重点消费 `need_review / review_focus / risk_level / require_data_review / allow_reasoned_fill`
- **尚书省 / 工部**：维护 `task_status / delivery_path / version / latest_change_note`

## 11. 一句话结论

《任务卡字段规范》的本质，不是做一个更复杂的表单，而是把公文任务从“口头描述”升级为“可起草、可审校、可流转、可追溯、可沉淀”的标准对象。

先把字段定清楚，三省六部才能真正协同；否则所有所谓流程，最后都会退化成临场补洞。
