# gongwen-assistant《工程包拆解与执行清单》

> 适用对象：围绕新项目 `gongwen-assistant` 的初版工程拆解、跨“省部”协同分工与落地推进。
>
> 当前判断：项目目录已创建，但尚无代码与详细设计沉淀；以下清单按“先搭骨架、再补能力、最后联调验收”的方式组织，便于直接进入执行。

## 一、项目目标

打造一个面向公文场景的智能助手，完成从**需求输入 / 公文起草 / 模板与规则约束 / 审核校验 / 工作流编排 / 交付留痕**的一体化能力闭环。首期优先聚焦：

1. 标准化公文生成与改写
2. 公文格式/规范校验
3. 多角色流转与状态可见
4. 可追溯的提示词、规则、产出物管理

## 三、分工程包执行清单

## WP-01 项目骨架与基础设施

- **负责省部**：工部
- **目标**：完成项目最小工程化落地，保证后续各包有统一的目录、依赖和运行方式。
- **依赖**：无
- **关键文件/模块**：
  - `projects/gongwen-assistant/README.md`
  - `projects/gongwen-assistant/package.json` 或 `pyproject.toml`
  - `projects/gongwen-assistant/.env.example`
  - `projects/gongwen-assistant/src/` 或 `app/`
  - `projects/gongwen-assistant/docs/architecture.md`
- **验收标准**：
  - 可以一键安装依赖与启动开发环境
  - 目录结构清晰，包含 docs / src / tests / examples 等基础分层
  - README 能说明项目目标、启动方式、目录说明
  - 配置项具备示例文件，敏感信息不入库
- **当前状态**：仅存在 `docs/` 目录，尚未形成完整骨架
- **下一步动作**：
  1. 确定技术栈（Node/TypeScript 或 Python）
  2. 初始化脚手架与基础目录
  3. 补齐 README、架构文档、环境变量模板

## WP-02 需求输入与任务建模

- **负责省部**：尚书省 + 吏部
- **目标**：明确系统要处理的“公文任务对象”，统一输入字段与业务约束。
- **依赖**：WP-01
- **关键文件/模块**：
  - `docs/requirements.md`
  - `docs/use-cases.md`
  - `src/domain/task.ts` / `src/domain/models.py`
  - `src/schemas/request.schema.json`
  - `examples/requests/`
- **验收标准**：
  - 定义清楚核心任务类型：起草、润色、改写、摘要、校核
  - 输入结构覆盖发文单位、文种、主题、对象、语气、篇幅、约束项等字段
  - 至少沉淀 5 个代表性场景样例
  - 模型/Schema 可被接口层与生成层复用
- **当前状态**：未开始，任务模型尚未定义
- **下一步动作**：
  1. 输出《需求清单与场景矩阵》
  2. 定义统一任务对象与字段字典
  3. 产出 5~10 个样例输入 JSON/Markdown

## WP-03 公文模板与知识规则库

- **负责省部**：礼部 + 门下省
- **目标**：沉淀公文模板、语体规范、格式规则与禁限项。
- **依赖**：WP-01、WP-02
- **关键文件/模块**：
  - `docs/templates.md`
  - `docs/style-guide.md`
  - `docs/policy-rules.md`
  - `prompts/templates/`
  - `knowledge/rules/`
  - `knowledge/examples/`
- **验收标准**：
  - 至少覆盖通知、请示、报告、函、会议纪要等核心文种
  - 每种文种具备结构模板、语气要求、字段要求
  - 形成可程序消费的规则清单（推荐 JSON/YAML）
  - 有明确的“必须项 / 禁止项 / 建议项”分类
- **当前状态**：未开始，规则库为空
- **下一步动作**：
  1. 先选 3~5 个高频文种做 MVP
  2. 为每种文种补模板、示例、约束规则
  3. 将规则整理为可供审核引擎消费的结构化文件

## WP-04 生成引擎与提示词编排

- **负责省部**：中书省
- **目标**：实现公文生成核心链路，包括起草、重写、续写、压缩与风格控制。
- **依赖**：WP-01、WP-02、WP-03
- **关键文件/模块**：
  - `src/prompts/`
  - `src/services/generation/`
  - `src/services/context-builder/`
  - `src/services/template-selector/`
  - `src/services/output-parser/`
- **验收标准**：
  - 能根据文种与输入参数自动选择模板与提示词
  - 支持至少 4 类能力：新起草 / 润色 / 改写 / 摘要
  - 输出结果结构稳定，可进入下一环节审核
  - 对失败、空输出、结构不完整等情况有兜底策略
- **当前状态**：未开始
- **下一步动作**：
  1. 设计 prompt 分层：系统规则 / 文种模板 / 用户输入 / 输出格式
  2. 建立模板选择器与上下文拼装器
  3. 先打通单轮生成 MVP

## WP-05 审核校验与风险控制

- **负责省部**：门下省 + 刑部
- **目标**：在生成结果后执行格式、逻辑、敏感项与合规风险检查。
- **依赖**：WP-03、WP-04
- **关键文件/模块**：
  - `src/services/review/`
  - `src/services/validators/`
  - `src/rules/review-rules.yaml`
  - `tests/review-cases/`
- **验收标准**：
  - 能输出结构化审核结果：通过 / 警告 / 拒绝
  - 至少覆盖格式缺项、称谓错误、语气不当、敏感表述、逻辑重复等问题
  - 校验结果可定位到段落或规则项
  - 对审核未通过结果给出修订建议
- **当前状态**：未开始
- **下一步动作**：
  1. 梳理审核规则优先级
  2. 先实现规则引擎 MVP（基于配置 + 程序校验）
  3. 形成一组正反例测试样本

## WP-06 工作流编排与状态流转

- **负责省部**：尚书省 + 太子
- **目标**：实现从提交任务到生成、审核、修订、完成的状态流转闭环。
- **依赖**：WP-02、WP-04、WP-05
- **关键文件/模块**：
  - `src/workflow/engine/`
  - `src/workflow/states.ts`
  - `src/workflow/actions.ts`
  - `docs/workflow.md`
- **验收标准**：
  - 定义清晰状态：待处理、生成中、待审核、待修订、已完成、已驳回
  - 支持失败重试、人工介入、回退再生成
  - 每一步有可记录的输入、输出、审计信息
  - 流程图与实现保持一致
- **当前状态**：未开始
- **下一步动作**：
  1. 先画流程图和状态机
  2. 定义事件、动作、回退条件
  3. 与接口层对齐任务生命周期字段

## WP-07 交付界面与对外接口

- **负责省部**：工部 + 户部
- **目标**：给用户或上层系统提供可调用入口，并控制成本与配额。
- **依赖**：WP-02、WP-04、WP-05、WP-06
- **关键文件/模块**：
  - `src/api/`
  - `src/cli/`
  - `web/` 或 `frontend/`
  - `docs/api.md`
  - `docs/ops-cost.md`
- **验收标准**：
  - 至少提供一种可用入口：HTTP API 或 CLI
  - 接口字段与任务模型保持一致
  - 有基本鉴权、限流、错误码说明
  - 成本/调用量可统计
- **当前状态**：未开始
- **下一步动作**：
  1. 明确首期交付形态（API 优先还是 Web 优先）
  2. 先做最小接口：创建任务、查询结果、获取审核意见
  3. 预留调用量与耗时埋点

## WP-08 测试验收与试点样例

- **负责省部**：尚书省 + 门下省
- **目标**：建立项目验收标准与回归样例，避免“能跑但不可用”。
- **依赖**：WP-03、WP-04、WP-05、WP-06
- **关键文件/模块**：
  - `tests/fixtures/`
  - `tests/e2e/`
  - `tests/regression/`
  - `docs/acceptance.md`
  - `examples/output/`
- **验收标准**：
  - 有覆盖核心文种和典型错误场景的样例集
  - 可执行回归测试，结果可比对
  - 形成明确的人工验收清单
  - 至少完成 1 轮试点样例评审
- **当前状态**：未开始
- **下一步动作**：
  1. 先定义验收维度：准确性、规范性、完整性、可读性、稳定性
  2. 建立黄金样例集与反例集
  3. 补齐 E2E 测试脚本
